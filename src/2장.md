## 1. 이름의 중요성
이름은 코드의 의미와 의도를 담는 설계 언어다.  
좋은 이름은 주석 없이도 코드의 목적을 명확히 전달한다.

```java
// ❌ 나쁜 예시
int d;

// ✅ 좋은 예시
int elapsedDaysSinceLastLogin;
```

---

## 2. 의도를 분명히 하라
이름만 보고 역할을 파악할 수 있어야 한다.

```java
// ❌ 나쁜 예시
List<int[]> getThem() { ... }

// ✅ 좋은 예시
List<Cell> getFlaggedCells() { ... }
```

---

## 3. 그릇된 정보와 혼동을 피하라
의미를 왜곡하거나 비슷한 이름을 사용하는 것은 혼란을 초래한다.

```java
// ❌ 나쁜 예시
Map<String, String> accountList; // 실제로는 리스트가 아님
int l = 1;  // 소문자 L
String O = "zero"; // 대문자 O

// ✅ 좋은 예시
Map<String, String> accountMap;
int lineCount = 1;
String orderId = "zero";
```

---

## 4. 의미 있게 구분하고 일관된 어휘를 유지하라
같은 개념은 같은 단어로, 다른 개념은 다른 단어로 표현하라.

```java
// ❌ 나쁜 예시
getUserInfo();
fetchUserData();
retrieveUserRecord(); // 같은 기능인데 단어만 다름

// ✅ 좋은 예시
getUser(); // 일관성 있게 ‘get’ 사용
```

---

## 5. 발음하기 쉽고 검색 가능한 이름을 써라
협업과 유지보수를 위해, 말로 설명하기 쉽고 검색이 용이한 이름을 사용하라.

```java
// ❌ 나쁜 예시
String genymdhms; // generation year month day hour minute second?

// ✅ 좋은 예시
String generationTimestamp;

int WORK_DAYS_PER_WEEK = 5;
```

---

## 6. 인코딩과 불필요한 접두어를 피하라
헝가리식 표기법이나 타입 인코딩은 현대 IDE 환경에서는 불필요하다.

```java
// ❌ 나쁜 예시
String m_strName;
IShapeFactory factory;

// ✅ 좋은 예시
String name;
ShapeFactory shapeFactory;
class ShapeFactoryImpl implements ShapeFactory { ... }
```

---

## 7. 명사와 동사의 구분
클래스 이름은 명사형, 메서드 이름은 동사형을 사용하는 것이 좋다.  
생성자 대신 의미 있는 정적 팩토리 메서드를 활용하라.

```java
// ❌ 나쁜 예시
class Calculate { ... }
void data() { ... }

// ✅ 좋은 예시
class Calculator {
    void calculate() { ... }
}

Number createFromRealNumber(double value) { ... }
```

---

## 8. 맥락을 부여하고 불필요한 맥락은 제거하라
모호한 변수에는 상위 구조를 통해 맥락을 부여하라.  
불필요한 접두어는 오히려 가독성을 해친다.

```java
// ❌ 나쁜 예시
String state; // 무슨 state인지 모름

// ✅ 좋은 예시
address.state;

// ❌ 나쁜 예시
GSDAccountAddress gsdAccountAddress;

// ✅ 좋은 예시
AccountAddress address;
```

---

## 9. 기발함보다 명료함을 택하라
유머나 은유적인 이름보다 직관적인 이름을 선택하라.

```java
// ❌ 나쁜 예시
deleteUserData(HolyHandGrenade);

// ✅ 좋은 예시
deleteUserData(DeleteItems);
```

---

## 10. 해법 영역과 문제 영역의 균형
해법 영역(기술적 개념)과 문제 영역(도메인 개념)을 구분해 적절히 혼합하라.

```java
// ✅ 좋은 예시
class AccountPolicyEvaluator {
    void applyPolicy(Account account) { ... }
}
```

---

## 11. 결론 ― 이름은 코드 품질의 거울
좋은 이름은 사고력과 언어 능력의 표현이다.  
명확한 이름은 협업과 유지보수를 용이하게 하고,  
코드의 의도를 장기적으로 보존한다.

```java
// ❌ 나쁜 예시
process();

// ✅ 좋은 예시
validateUserCredentials();
```
